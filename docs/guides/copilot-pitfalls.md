# VS Code Copilot Chat 为什么这么难用

如果你在 VS Code 里主要使用的是 Copilot Chat，那么真正决定体验上限的，通常不是补全质量，而是你有没有把 `Ask`、`Plan`、`Agent` 这三种模式用对。

其中最容易翻车的，是 `Plan` 和 `Agent`。

`Plan` 的职责是先把问题想明白，形成可执行步骤；`Agent` 的职责是根据目标自主读代码、改文件、跑命令、验证结果。如果你让 `Plan` 去拍脑袋定方案，或者让 `Agent` 在没有边界的前提下直接开工，结果往往都不会太好。

这份文档不打算替它洗地，而是直接拆三件事：

- 它具体烂在哪里
- 这些问题为什么会发生
- 在它就这水平的前提下，怎么让它尽量正常工作

### 1. 模式很多，但边界不够硬

理论上有 `Ask`、`Plan`、`Agent` 三种模式，听起来很完整；但真实使用里，这三者的职责边界对普通用户并不总是足够清晰。

常见体感是：

- `Plan` 像在规划，但其实已经带入了实现假设
- `Agent` 像在执行，但经常夹带自己的问题定义
- 用户以为自己在“切模式”，实际上只是换了一个更会主动行动的聊天框

### 2. 会话上下文太容易脏

VS Code Copilot Chat 最大的问题之一，是很容易把调研、规划、执行、追问混在一个 session 里，最后谁也分不清当前回答到底继承了哪些旧上下文。

- 旧约束容易残留
- 已经被否定的方案还在反复出现
- 当前任务会被前一段对话污染

这类问题不是“模型笨”，而是产品交互默认鼓励你一直聊下去，而不是及时切 session、切阶段、切上下文。

### 3. Plan 看起来完整，但不一定真能落地

Copilot Chat 的 `Plan` 常常能产出一份形式上很完整的计划：有步骤、有顺序、有验证。但它经常会出现一个问题：计划写得像回事，执行时却发现前提没锁住。

典型表现：

- 没先确认接口能不能改
- 没确认测试命令是否真实可用
- 没确认依赖、脚本、目录约束是否存在

所以它的问题不是“不会写计划”，而是“计划文档感很强，工程约束感不够强”。

### 4. Agent 自主性上来了，但可预测性没有同步上来

一旦进入 `Agent` 模式，VS Code Copilot Chat 会显得比普通 chat 主动很多：读文件、跑命令、改文件、继续修。但这个“主动”，并不总能带来“稳定”。

最常见的问题是：

- 改动范围比你预期大
- 在错误理解上持续加码
- 没有先锁边界，就开始施工
- 验证动作做了，但验证目标不对

也就是说，它不是不能自动执行，而是“自动执行的过程”比“自动执行的结果”更值得被怀疑。

### 5. 调试入口有了，但主路径依然不够顺

官方现在已经补了不少调试能力，比如日志、`Collect Diagnostics`、Agent Debug Log、Chat Debug View。这些当然有用，但它们更多像是出事后的补救手段，而不是把主路径本身打磨得足够稳。

这也是很多人会觉得它很烦的原因：

- 不是完全不能排障
- 而是你排障的频率偏高
- 你会更频繁地需要确认“它为什么这么想、为什么这么做”

一个工具如果经常需要靠 debug 面板来理解它的基本行为，那说明它的正常使用心智模型并不够自然。

### 6. UI 集成很强，但工程闭环不总是强

VS Code 原生集成当然是 Copilot Chat 的最大优势：编辑器、终端、面板、会话、扩展生态都在一个地方。这让它“看起来”像最顺手的 agent 入口。

但问题也在这里。UI 集成强，不等于工程闭环强。很多时候你获得的是：

- 很方便地下达任务
- 很方便地看它改文件
- 但不一定很方便地保证这件事按你的真实约束做完

所以它常给人的感觉不是“能力缺失”，而是“最后 20% 的可控性不够，导致前面 80% 的顺滑都被抵消”。

### 更尖锐一点的结论

VS Code Copilot Chat 的问题可以概括成一句话：

> 它已经有了 agent 的壳、agent 的入口、agent 的按钮，但很多时候还没有把 agent 最关键的东西做好：稳定边界、干净上下文、可预测执行。

所以用户会觉得它“烂”，往往不是因为它一次都做不成事，而是因为：

- 成功时像运气不错
- 失败时像理所当然
- 纠偏成本比预期高

## 先记住两个边界

### `Plan` 不是架构师替身，它更像“会写文档的实习生”

`Plan` 会研究代码、列步骤、列验证项，但它不会自动替你做正确的产品决策和技术决策。你不给约束，它就会默认一套“看起来合理”的方案。问题在于，Copilot Chat 的 `Plan` 往往更擅长把方案写得顺，而不是把前提锁得死。

### `Agent` 不是稳定的自动施工队，它非常容易“先干再说”

`Agent` 可以自己搜上下文、改多个文件、调用工具、执行终端命令，但这不等于它会天然理解你的真实目标。它只是在当前上下文里，尽力完成它以为你要做的事。Copilot Chat 很容易给你一种“已经开始推进了，但方向未必对”的体感。

如果你没有明确下面三件事，`Plan` 和 `Agent` 都容易开始胡来：

- 要达成什么结果
- 不允许改什么
- 用什么标准验证完成

## Plan 模式相关

### 坑 1：把 Plan 当成“先随便出个方案看看”

很多人进入 `Plan` 模式后，给一句模糊要求，例如“优化登录系统”或者“重构一下这块代码”，然后拿到一个看起来条理清楚的计划，就默认它已经正确理解了问题。

这是最典型的误判。计划写得顺，不代表问题定义对。Copilot Chat 特别容易产出一份“像正式方案”的文本，让你误以为问题已经被分析清楚。

常见后果：

- 把“局部修复”规划成“整体重构”
- 把“补测试”规划成“顺手改实现”
- 把“定位 bug”规划成“先做架构调整”

规避方式：

- 在 prompt 里写清楚任务类型：排障、重构、补测试、功能开发，不要混着说
- 明确边界，例如“只允许改认证模块，不要调整数据库 schema”
- 明确输出要求，例如“先给计划，不要写代码”

### 坑 2：Plan 没有澄清关键前提，你也继续往下走

官方流程本身就假设 `Plan` 会先研究任务并提出澄清问题。如果它没有问到关键前提，你也没有主动补，后面的计划通常建立在错误假设上。Copilot Chat 很容易跳过这些前提确认，直接给你一个“能往下聊”的方案。

高风险前提包括：

- 是否允许改接口
- 是否允许升级依赖
- 是否需要兼容旧数据或旧行为
- 测试、构建、部署哪个才是验收标准

规避方式：

- 把缺失前提直接写在 prompt 里
- 明确要求它“如果信息不足先提问，不要直接规划”
- 拿到计划后，先检查假设列表，再看步骤列表

### 坑 3：只看步骤，不看验证方案

一个真正有用的 `Plan` 不只是列出“先干什么、后干什么”，还应该说明每一步怎么验证。没有验证标准的计划，交给 `Agent` 后就会变成“不断改动直到看起来差不多”。Copilot Chat 经常能把步骤说清楚，但没把“怎么证明做对了”说清楚。

规避方式：

- 要求计划里单独列出 verification steps
- 要求它写清楚使用什么命令、看什么输出、哪些结果算通过
- 如果计划没有验收标准，先补计划，不要立刻 handoff 给 `Agent`

### 坑 4：以为 Plan 是长期记忆

`Plan` 会把计划保存到当前 session 的 `plan.md` memory 里，但它不是项目级长期知识。会话结束后，这份 plan 不会自动成为未来所有会话的共享上下文。Copilot Chat 这里很容易让人误以为“它应该记得”。

规避方式：

- 重要计划及时落到仓库文档、issue 或设计稿里
- 不要假设下一个 session 会天然继承本次规划结论
- 需要长期复用的规则，写到 instructions，而不是留在一次性 plan 里

## Agent 模式相关

### 坑 5：把 Agent 当成加强版自动补全

`Agent` 适合处理多文件、需要检索上下文、需要跑命令和验证结果的任务；它不适合拿来做一句话都说不清的小修小补。

Copilot Chat 在这个坑上尤其明显，因为它就在 VS Code 里，入口离普通 Chat、补全、Inline Chat 太近，很多人自然会把它当成“更主动一点的聊天框”，而不是一个需要任务设计和边界控制的执行器。

常见错误：

- 直接说“帮我优化一下这个项目”
- 直接说“把这个系统重构一下”
- 没有说明限制条件，就让 Agent 自己决定架构、依赖和目录结构

后果通常是：

- 改动范围失控
- 自作主张新增依赖
- 在错误的模块上持续用力

规避方式：

- 小修改优先用 Ask 或 Inline Chat
- 多文件任务先让 Plan 出步骤，再决定是否执行
- 明确写出边界，例如“只改 `src/auth`，不要动 API 协议，不要升级依赖”

### 坑 6：没过 Plan，就直接让 Agent 开工

复杂任务如果一开始就进入 `Agent` 模式，Copilot 很容易在错误理解上越跑越远。尤其是迁移、重构、补测试、排查线上问题这类任务，先做计划几乎总是更便宜。Copilot Chat 的 UI 很容易让你跳过这一步，直接开始动手。

规避方式：

- 先 Ask：让它解释现状、读关键文件、总结依赖关系
- 再 Plan：要求它拆出步骤、列风险、列验证方式
- 最后 Agent：只在计划足够具体时执行或 handoff

一个实用提示词模板：

> 先不要改代码。先阅读相关文件，给出实施计划、风险点和验证步骤；如果信息不足，先提问。

### 坑 7：权限开太大，却没有监督点

`Agent` 的权限级别越高，执行效率越高，但失控代价也越高。尤其在 Bypass Approvals 或更激进的自动化模式下，它可能连续调用工具、修改多个文件、跑一串命令，然后把错误路径越走越深。Copilot Chat 在这里的问题，不是没有权限系统，而是权限一旦放开，执行过程本身未必足够可预测。

规避方式：

- 不熟悉仓库时，先用更保守的 approvals
- 第一次执行某类任务时，先观察它会读哪些文件、跑哪些命令
- 把关键限制写进 prompt，而不是只依赖权限弹窗兜底

### 坑 8：把长任务扔给 Agent 后就不管了

Agent 会连续做很多假设。你不及时纠偏，它就会把这些假设继续放大。结果不是“一次做完”，而是“一次做偏”。Copilot Chat 往往表面上显得很顺，实际上已经在错误路径上持续推进。

规避方式：

- 在第一个阶段就检查它准备修改哪些文件
- 看它是否误读了模块边界或接口约定
- 一旦方向不对，尽早停止并重新发 prompt，不要寄希望于后续自动修正

### 坑 9：以为 Agent 会自动知道你想保守修改

如果你不明确写“优先最小改动”“不要改接口”“不要新增依赖”，`Agent` 经常会为了完成目标，顺手扩大改动面。Copilot Chat 很容易出现“先把事情做成，再说代价”的倾向。

规避方式：

- 在 prompt 里写“优先最小变更”
- 明确禁止项：不要重命名公共 API、不要改 schema、不要动 CI 配置
- 要求它先说明打算修改哪些文件，再开始执行

## 上下文与会话相关

### 坑 10：给太少上下文，然后怪它不懂项目

很多人只丢一句“按项目风格实现”，但项目本身没有指令文件、没有清晰约束、没有告诉 Copilot 哪些模块是核心。它当然只能猜。Copilot Chat 很容易让人产生一种错觉：既然它就在编辑器里，那它应该天然懂项目。

规避方式：

- 用 `/init` 生成项目级 instructions
- 在指令里写清楚架构原则、代码风格、测试命令、禁止事项
- 提示时显式引用文件、目录或符号，而不是让它盲猜

### 坑 11：把 Plan、Ask、Agent 混在一个脏会话里

另一类常见问题是一个聊天窗口里连续混入多个不相关任务：先问配置，再问架构，再问线上 bug，最后再让它写代码。上下文一旦污染，后续回答质量会明显下降。Copilot Chat 在这点上尤其吃亏，因为它的主交互天然鼓励“继续在这个聊天里聊下去”。

典型症状：

- 明明在修测试，却开始引用前面聊过的部署细节
- 把已经失效的约束当成当前要求
- 反复围绕旧方案打转

规避方式：

- 不相关任务开新 session
- 长对话定期压缩或重开
- 调研型问题和执行型问题分开处理

### 坑 12：以为它“自动知道”终端、报错和运行状态

Copilot 可以使用终端输出、失败日志、源码索引这些上下文，但前提是你把相关信息暴露给它，或者它当前模式确实能拿到这些上下文。Copilot Chat 很容易让用户误判“它应该已经看到这些了”。

规避方式：

- 报错类问题直接附终端输出或测试失败信息
- 明确说“根据当前失败日志定位原因”
- 不要只说“这个没跑起来，你修一下”

## 安全、权限与排障

### 坑 13：把敏感信息直接塞进 prompt

无论是 Chat、Agent 还是第三方 MCP，默认都不值得你把真实密钥、客户数据、生产库连接串直接贴进去。AI 工具的便利性不应该建立在泄露敏感信息之上。Copilot Chat 因为深度长在编辑器里，反而更容易让人降低警惕，把“正在本地 IDE 里操作”误解成“信息天然更安全”。

规避方式：

- 用脱敏数据、伪造样本、占位符替代真实数据
- 明确哪些目录、文件或变量不允许被读取和上传
- 对涉及公司代码、隐私数据、合规要求的场景，先确认组织策略

### 坑 14：让 Agent 做完就算完，不看日志和调试视图

Copilot Chat 在 VS Code 里已经提供了足够多的排障入口。如果 `Plan` 的推理有问题，或者 `Agent` 的执行顺序不对，继续补 prompt 往往不如先看调试信息。问题在于，它更像是“出了事以后你还有地方查”，而不是默认就把主路径做稳。

至少值得记住这些入口：

- 把 GitHub Copilot 和 GitHub Copilot Chat 的日志级别调到 `Trace`
- 查看 `GitHub Copilot` / `GitHub Copilot Chat` 输出面板
- 使用 `GitHub Copilot: Collect Diagnostics`
- 在 Chat 中查看 Agent Debug Log
- 在 Chat 中查看 Chat Debug View

## 排障建议

如果你已经觉得 VS Code Copilot Chat 的 `Plan` 或 `Agent` “开始发癫”，优先按这个顺序排查：

1. 任务是不是太模糊，边界没写清楚。
2. 当前入口是不是选错了，例如该用 `Plan` 却直接用了 `Agent`。
3. `Plan` 有没有确认关键前提，还是直接带着假设出方案。
4. `Agent` 有没有在没有约束的情况下扩大改动范围。
5. 会话是否已经被无关上下文污染。
6. 项目是否缺少 instructions、文件引用或测试期望。
7. 是否应该打开日志和调试视图，而不是继续盲聊。

## 实用结论

VS Code Copilot Chat 里最常见的误区，不是模型不够强，而是它比不少同类 agent 更容易把“规划、执行、上下文、验证”搅在一起，导致用户必须自己承担更多纠偏工作。

更靠谱的顺序通常是：

- 用 `Ask` 搞清楚现状
- 用 `Plan` 锁定步骤、风险和验证方式
- 用 `Agent` 执行边界明确、可验证的多文件工作
- 永远把日志、测试和人工 review 留在最后一道关
